# Калькулятор прибыли WB — модель данных и открытые вопросы

Документ для согласования с командой. Стек намеренно не зафиксирован — модель логическая (СУБД-агностичная). Решения, принятые на текущий момент, помечены ✅.

## Принятые решения
- ✅ **Стек**: Flask + PostgreSQL + SQLAlchemy/psycopg + Redis + Pydantic v2. Локальные дампы — `.dump`.
- ✅ **ИЛ = 1, ИРП = 0** — захардкожены в конфиг (источника значений из WB пока нет).
- ✅ **Только Короб / Монопаллета** — СГТ / КГТ / суперсейф вне MVP.
- ✅ Лимит расчётов на пользователя — в конфиг (по ТЗ 10).

## 2. Логическая модель (сущности)

### Пользовательские (изменяемые)
- **User** — `id`. Связь 1—N с Calculation (≤ лимита).
- **Calculation** — `id`, `userId`, `name`, `position`, и параметры:
  - товар: `categorySubject` (ссылка на предмет), `priceRub`, `length/width/height` (см), `weightKg`/`isHeavy`
  - `salesModel` ∈ {FBS, FBW, DBS, EDBS, C&C}
  - `deliveryType` ∈ {box, monopallet} — только для FBW
  - `warehouseIds` (1–5)
  - статистика: `turnoverDays`, `buyoutPercent` (1–100)
  - расходы: `costPrice` (≥0), `promoPercent`, `otherExpensesPerUnit`
  - налоги: `taxSystem` ∈ {usn_income, usn_income_minus_expenses, other, none}, `taxRate` (обязат. при `other`)
  - **истории нет** — каждый пересчёт перезаписывает результат; автосохранение параметров при каждом изменении.

### Справочные (правит админ, версионируются датами — см. Q3)
- **CommissionRef** (category, subject, commission по моделям) — дерево, плоский список с путём, **кэш 1 час**.
- **Warehouse** (id, name, active, коэффициенты, supportedTypes) — неактивные не показывать, но старые расчёты не ломать.
- **BaseTariff / StorageTariff** — `effectiveFrom` / `effectiveTo`, старые не удалять, закрывать датой.

### API расчёта (ответ)
`{ calculation, result.byWarehouse[] }`, каждый элемент: `warehouseName, priceRub, wbCosts.*, transferToSeller, taxes, costPrice, totalExpenses, profit, marginality, roi` — в ₽ и %. ROI = null при себестоимости 0. Ошибка тарифа — на уровне колонки.

**Состав `wbCosts` (Шаг 4 + находка C9):**
`commission` + `acquiring` (1.5%) + `logistics` + `storage` + `acceptance` (приёмка — C9, дефолт 0) = `wbCostsTotal`.

## 3a. Цена безубыточности (C4 — точное решение)

Цена `P`, при которой `Прибыль(P) = 0`. Обозначения на единицу товара:
- `c = (комиссия% + эквайринг%)/100` — переменная доля удержаний WB (растёт с ценой)
- `F = логистика + хранение` (₽, фикс., т.к. ИРП=0)
- `CP` — себестоимость, `O` — прочие расходы (₽, фикс.)
- `pr = продвижение%/100`, `t = ставка_налога/100`

Закрытые формулы по системе налогообложения (проверены бисекцией, Δ=0):

| Система | Формула P_безубыт |
|---|---|
| УСН «Доходы» / «Другая» | `(F·(1−t) + CP + O) / ((1−c)·(1−t) − pr)` |
| УСН «Доходы−расходы» | `(1−t)·(F + CP + O) / ((1−c)·(1−t) − pr)` |
| «Не учитывать» (t=0) | `(F + CP + O) / (1 − c − pr)` ← совпадает с формулой ТЗ |

**В коде:** реализовать через **бисекцию** по `profit(P)` (робастно к излому УСН «Д−Р», где налоговая база может стать ≤0, и к делению на ≤0). Закрытые формулы — для тестов/быстрой проверки. Если знаменатель ≤ 0 (переменные доли ≥ 100%) → безубыточности нет, вернуть null.

## 3b. Версионность тарифов (C3)

- Каждая строка тарифа (логистика/хранение/комиссия): `effective_from DATE NOT NULL`, `effective_to DATE NULL`.
- Интервалы **полуоткрытые** `[effective_from, effective_to)` — без пересечений на стыке.
- **Первый импорт (сид):** `effective_from = 2020-01-01`, `effective_to = NULL` (якорь в прошлом → любая дата выбора находит тариф).
- **Выбор в MVP:** дата = «сегодня» (CURRENT_DATE):
  `WHERE effective_from <= :date AND (effective_to IS NULL OR effective_to > :date)` → при нескольких берём `MAX(effective_from)`.
- **Обновление админом** (не удаляем, закрываем):
  ```
  UPDATE ... SET effective_to = :new_from WHERE ... AND effective_to IS NULL;  -- закрыть текущий
  INSERT ... (effective_from = :new_from, effective_to = NULL);                -- открыть новый
  ```
- Задел: если позже добавим «дату заказа»/историю — схема уже поддержит выбор тарифа на любую прошлую дату.
