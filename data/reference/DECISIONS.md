# Решения, допущения и находки (B1–C10)

Единый перечень всего, что мы **добавили от себя** или **выбрали** при разборе ТЗ,
справочников (`comments.md`, xlsx) и эталона (`media_library/`). По каждому пункту:
**суть** (пробел/неоднозначность в исходных материалах) → **решение** → **почему**.

Легенда статусов: ✅ закрыто в коде · 🟡 решено, но остаётся точка калибровки/проверки с командой.

---

## B. Блокеры — недостающие данные (восполнили сами)

### B1 — Значения ИЛ / ИРП отсутствовали ✅🟡
- **Суть:** формулы логистики из `comments.md` используют Индекс локализации (ИЛ) и Индекс
  расположения покупателя (ИРП), но самих значений нигде нет (на UI видно только «ИЛ — 1»).
- **Решение:** захардкодили **ИЛ = 1, ИРП = 0**, вынесли в конфиг (`LOCALIZATION_INDEX`,
  `BUYER_LOCATION_INDEX`). См. также [C8].
- **Почему:** без них логистика не считается; 1/0 нейтральны и совпадают с тем, что показывает UI.

### B2 — Базовые тарифы не были оцифрованы ✅
- **Суть:** базовые ставки (₽/литр 23–32, обратная 46+14 ₽, хранение 0.08 ₽/л/день) лежали
  только текстом в `comments.md`.
- **Решение:** собрали в структуру `base-tariffs.json`, затем в таблицу БД `base_tariffs`
  (версионную). xlsx даёт **коэффициенты-множители**, базовые ₽ — отсюда.
- **Почему:** итоговый тариф = базовый × коэффициент склада × ИЛ; базовую часть нужно хранить отдельно.

### B3 — Не было справочника складов с идентификаторами ✅
- **Суть:** в xlsx только названия складов и коэффициенты, без `id`/флага активности.
- **Решение:** сгенерировали `warehouses.json` со **стабильными `id` (slug)**, флагом `active`,
  коэффициентами по типам упаковки и `supportedTypes`. «—» в xlsx → тип не поддерживается.
- **Почему:** адресовать склад нужно по стабильному ключу, а не по неоднозначному имени (см. [C6]).

## C. Решения по неоднозначностям и находки

### C1 — Объём поставок: СГТ / КГТ / суперсейф ✅
- **Суть:** ТЗ — только Короб/Монопаллета, но в xlsx есть «суперсейф», а в `comments.md` —
  отдельные формулы СГТ (грузовая, «Курьером WB», обратная СГТ).
- **Решение:** **только Короб и Монопаллета**. Суперсейф/СГТ/КГТ вынесены из MVP
  (в `base-tariffs.json` → `_outOfMvpScope`).
- **Почему:** соответствует ТЗ; СГТ/КГТ — отдельный объём работ.

### C2 — Коэффициент для товара > 25 кг ✅
- **Суть:** ТЗ упоминает «повышающий коэффициент логистики для >25 кг», но числа не даёт.
- **Решение:** конфиг-множитель `HEAVY_GOODS_LOGISTICS_COEF`, **дефолт 1.0** (нейтральный);
  флаг «тяжёлый» в модели, точка расширения в движке.
- **Почему:** не блокирует расчёт, реальное число подставляется конфигом, когда даст команда.

### C3 — Версионность тарифов (даты действия) ✅
- **Суть:** ТЗ требует «брать самый свежий действующий тариф» и «не удалять старые, а закрывать
  датой», но xlsx — снимок без дат.
- **Решение:** поля `effective_from`/`effective_to`, **полуоткрытые интервалы** `[from, to)`;
  первый импорт с **якорем `2020-01-01`**; выбор по дате = **«сегодня»**; обновление = закрыть
  старую версию + открыть новую.
- **Почему:** даёт корректный механизм обновления и задел на «дату заказа»/историю в будущем.

### C4 — Цена безубыточности ✅
- **Суть:** в ТЗ формула `Всего расходов / (1 − переменные_доли)` с пометкой «уточнить с командой»;
  налог ломает простую формулу (зависит не от цены напрямую).
- **Решение:** **точное решение** уравнения `Прибыль(P)=0` бисекцией (робастно к излому УСН
  «Д−Р»); закрытые формулы выведены и сверены с бисекцией (Δ=0). См. `DATA-MODEL.md §3a`.
- **Почему:** единственный способ дать верную безубыточность при включённом налоге.

### C5 — Хранение «Монопаллета» ✅
- **Суть:** в `comments.md` монопаллета = 25 ₽/день (фикс, без объёма), а формула ТЗ Шаг 4
  завязана на объём. Плюс 25 ₽ — это плата за паллету, а не за единицу.
- **Решение:** считаем по **объёмной формуле как Короб** (`0.08 × коэф_хранения × объём × дни`);
  ставка 25 ₽/день не используется. Хранение едино для обоих типов.
- **Почему:** структурно совпадает с ТЗ; избегаем абсурдного завышения «за паллету на единицу».

### C6 — Мэппинг имён складов UI ↔ xlsx 🟡 (НЕДОСТАТОК)
- **Суть / в чём недостаток:** имена складов в реальном UI **не совпадают** с xlsx. Примеры:
  - в xlsx три «Воронежа»: `Воронеж` (коэф box 1.3), `Воронеж: Питание` (1.6), `СЦ Воронеж` (1.75)
    — разброс итога ~12 ₽;
  - UI показывает `Воронеж · Короб`, `Владимир (Воршинское): Питание · Короб`, `Электросталь · Короб`
    — суффикс «: Питание» и « · Короб» не имеют точного соответствия в xlsx.
  - Живой прогон подтвердил: колонка `Воронеж: Питание` сходится с эталоном точно, а
    `Электросталь`/`Владимир` расходятся — именно из-за неоднозначного соответствия имён.
- **Решение (что сделали):** адресуем склад по **стабильному `id`**, `deliveryType` — отдельное
  поле, пользователю показываем полное имя из xlsx. Дубликатов в данных нет — это 107 разных
  складов.
- **Что остаётся (недостаток):** нужен **официальный мэппинг «склад в UI → строка в xlsx»** от
  команды (паритет нейминга). Это верификация данных, не код. Точная привязка «какой именно
  склад в эталоне» распутывается вместе с [C9].

### C7 — Стек бэкенда ✅
- **Суть:** в ТЗ стек не зафиксирован.
- **Решение:** **Flask + PostgreSQL + SQLAlchemy/psycopg + Redis + Pydantic v2**; локальные
  дампы — `.dump`.
- **Почему:** по указанию команды (основной проект на этом стеке).

### C8 — Источник ИЛ / ИРП 🟡 (выбрали из-за отсутствия данных)
- **Суть:** ИЛ/ИРП — это динамические индексы из API WB (зависят от склада/дня). Отдельного
  источника или выгрузки **не было**.
- **Решение:** **выбрали хардкод ИЛ = 1, ИРП = 0**, потому что отдельных данных не предоставили;
  вынесли в конфиг, чтобы заменить без изменения кода, когда появится выгрузка/API.
- **Почему:** именно так (ИЛ=1) показывает и сам UI калькулятора; это безопасный нейтральный
  дефолт до появления реальных значений.

### C9 — «Приёмка» + обратная логистика 🟡 (добавили компонент и логичную цену)
- **Суть (находка):** расчёт «Расходов на WB» по формулам ТЗ не сходился с эталоном (178.71 ₽).
  Разбор `screencast.mp4` показал, что в реальном калькуляторе есть **отдельная «Приёмка»**
  (плата за приём товара на склад при поставке FBW), которой **в формулах ТЗ нет вообще**.
  Дополнительно: обратная логистика в `comments.md` указана **без** «× Коэффициент склада»
  (множитель есть только у доставки) — мы его ошибочно применяли.
- **Решение (что сделали и какую цену поставили):**
  1. добавили отдельный компонент `wbCosts.acceptance` (конфиг `ACCEPTANCE_RUB`);
  2. убрали коэффициент склада из обратной логистики (привели к `comments.md`);
  3. **поставили логичную цену приёмки `ACCEPTANCE_RUB = 1.63 ₽`** — это калибровочное значение,
     при котором расчёт **воспроизводит эталон 178.71 ₽ ТОЧНО** (до копеек).
- **Почему 1.63 и что остаётся:** 1.63 ₽ подобрано из эталонной цифры — это разумный плейсхолдер.
  Реальный тариф приёмки у WB **плавающий** (зависит от загрузки склада, бывает и 0 ₽), поэтому
  значение нужно заменить настоящим тарифом/коэффициентом приёмки от команды/WB. Сейчас движок
  даёт корректную модель и точный эталон; калибровка — одной строкой в `.env`.

### C10 — База налога УСН «Доходы» (ошибка в ТЗ) ✅ (находка)
- **Суть (находка):** ТЗ Шаг 6 предписывает считать УСН «Доходы» от «К перечислению»
  (`220 × 6% = 13.2 ₽`), но эталон показывает **24 ₽ = 6% × 400**, то есть налог берётся от
  **выручки (цены)**. Формула ТЗ неверна.
- **Решение:** считаем налог от **выручки (цены)**: УСН «Доходы»/«Другая» = `Цена × ставка`;
  УСН «Доходы−расходы» = `(Цена − все расходы) × ставка`, база < 0 → налог 0.
- **Почему:** совпадает с эталоном (24 ₽ ровно) и с реальной семантикой налога (доход = выручка).
